Google SSO + allowlist is mandatory

VPN is an additional production control

Require private ingress for cloud-hosted production administration where operationally feasible. Do not use VPN membership as a substitute for identity, allowlisting, RBAC, or step-up approval.

Product requirements

Detailed product, architecture, security, and delivery definition.

BoxLite Deploy Admin Management — PRD

Status: Proposed · Audience: Product, Platform, Security, SRE, Application Engineering · Language: English

Product decision: Treat apps/infra-local and apps/infra as execution adapters, not as the Deploy Admin application itself. The management plane owns intent, policy, approvals, evidence, and orchestration; short-lived Kubernetes Jobs invoke the existing adapters.

1. Problem and goals

BoxLite deployment is currently initiated through repository commands. This makes repeatability, least privilege, approvals, visibility, and evidence depend on individual operator practice. Deploy Admin must provide one governed experience from commit selection through build, deployment, verification, and recovery.

Goals

  • Manage local BoxLite through local Kubernetes and the infra-local adapter.
  • Manage remote staging through a local Kubernetes-hosted control plane and the infra SST/AWS adapter.
  • Package the same Deploy Admin release as a signed image and YAML/Helm installation for cloud Kubernetes, from which remote BoxLite environments can be managed.
  • Unify build provenance, deployment state, approvals, logs, health verification, audit evidence, and RBAC.
  • Support Google SSO with explicit admission; optionally require VPN/private ingress.

Non-goals

  • Rewriting SST, Pulumi, BoxLite SDK orchestration, Make targets, or CI build logic.
  • Running untrusted arbitrary shell supplied from the browser.
  • Making a browser process hold AWS, Kubernetes, registry, Cloudflare, or OIDC secrets.
  • Automating protected Runner replacement without its explicit drain and recovery ceremony.

2. Current-state evidence and constraints

Repository factProduct implication
infra-local drives 11 BoxLite microVM services and 4 native host processes; its CLI exposes up, down, status, logs, restart, reset, and nuke.The local adapter initially supports these exact operations. Kubernetes packaging does not magically make native macOS processes portable; MVP requires a privileged/local worker attached to the Mac, while a later container-native topology must be separately engineered.
infra deploys AWS through SST v4 and a Cloudflare-aware wrapper.The remote adapter must invoke repository wrappers, obtain short-lived AWS identity, load per-stage Cloudflare credentials, capture an SST diff, and retain deploy evidence.
The cloud stack uses external standards-compliant OIDC; the dashboard uses OIDC redirect login. The documented cloud example is Auth0 and local development uses Dex; Google-specific provider configuration is not present.Reuse the generic OIDC protocol and redirect pattern. Google IdP setup, a separate Deploy Admin client/audience, and the authorization store are new work.
There are no Kubernetes manifests for Deploy Admin in the current repository.Kubernetes installation, job isolation, network policy, service account, secret references, persistence, and ingress are new deliverables—not existing capabilities.
Runner instances contain workload state and are protected from routine replacement.Any IaC plan that replaces or deletes a Runner is blocked by policy and needs a dedicated, approved recovery workflow.

3. Users and authorization

RoleCapabilities
ViewerRead environments, runs, logs with secret masking, health, and audit records.
OperatorRun non-production build/deploy workflows, retry safe stages, and execute approved operational actions.
ApproverApprove or reject protected environment releases; cannot approve own production request.
Environment AdminVersion environment configuration, bind credentials, set policy, and manage environment-scoped operators.
Platform OwnerManage administrator allowlist, global policy, identity settings, retention, and break-glass access.
AuditorRead and export immutable evidence; no execution access.

Authorization is deny-by-default and evaluated server-side on every request and worker action. Production uses separation of duties. Break-glass access is time-bound, requires a reason, alerts owners, and is fully audited.

4. Functional hierarchy

L1 — Portfolio overview

  • Environment readiness, active releases, build success, lead time, recent failures, and actionable alerts.
  • Freshness timestamp and explicit unavailable/stale states; no fabricated health from client cache.

L2 — Source and build

  • Select an allowlisted repository, immutable commit SHA, and predefined pipeline.
  • Run repository-owned Make/CI targets in an isolated Job; stream structured logs; enforce timeout and cancellation.
  • Record source SHA, dependencies, test summary, SBOM, vulnerability policy, artifact digest, signature, and provenance.
  • Artifacts are promoted by digest across environments; production is never rebuilt from a mutable branch.

L3 — Environments and configuration

  • Environment types: local infra-local, remote staging infra, and remote production infra.
  • Versioned YAML schema with repository ref, adapter, stage, region, domains, artifact policy, credential references, verification policy, approval policy, and retention.
  • Draft → validate → diff → approve → activate. Secrets are referenced by name/version and never displayed or stored in Git.
  • Preflight validates worker reachability, Kubernetes capacity, repository access, credentials, AWS identity, SSM keys, Docker availability where required, OIDC discovery, DNS prerequisites, and environment lock.

L4 — Deployment orchestration

Create immutable run→Acquire environment lock→Preflight→Build / resolve artifact→Generate diff→Policy + approval→Deploy→Verify→Close evidence
  • Only one mutating run per environment. Read-only checks may execute concurrently within bounded quotas.
  • Cancellation is cooperative and stage-aware. It never labels partially applied infrastructure “cancelled and safe” without reconciliation.
  • Retry creates a new attempt linked to the original run and resumes only where the adapter declares idempotency.
  • Failure shows stage, sanitized cause, affected resource, retry safety, remediation, and links to logs/evidence.

L5 — Verification and recovery

  • Local: adapter status is healthy; L1 and L2 required services pass; API and dashboard respond; Runner is ready.
  • Remote: SST completes; API health passes; dashboard assets load; at least one Runner is ready; managed smoke scenario passes and cleans up.
  • Rollback means artifact/config redeployment when supported. Infrastructure rollback is always a new reviewed diff—not a blind inverse command.
  • Destructive local reset/nuke and cloud remove/replacement require typed environment name, reason, recent re-authentication, policy approval, and retained evidence.

L6 — Observability and audit

  • Structured run events, live logs, duration by stage, queue wait, success rate, rollback rate, deployment frequency, and verification latency.
  • Secret masking happens before persistence and streaming. Raw credentials, tokens, environment files, and signed URLs are forbidden in logs.
  • Audit records cover login allow/deny, role/allowlist changes, configuration diffs, credential reference changes, run creation, approvals, commands selected from the fixed catalog, results, exports, and break-glass actions.

6. Authentication evaluation

OptionStrengthsLimitationsDecision
Google SSO + admin allowlistNamed identity, lifecycle, MFA at IdP, explicit product admission, works for local and cloud.Public endpoint remains attackable; allowlist administration needs bootstrap and recovery.Mandatory baseline. Validate issuer, audience, signature, expiry, hosted domain where applicable, and normalized verified email. Allowlist by stable IdP subject plus email for display.
VPN + Google SSOReduces exposed surface and adds device/network posture.VPN is not user authorization; operational overhead; can impede incident access and automation.Recommended additional control for cloud production. Combine with SSO, allowlist, RBAC, and approval. Local development may use loopback/private ingress.

On first successful OIDC callback, deny access unless an active allowlist entry matches the stable identity. Never auto-provision a Deploy Admin role from ordinary BoxLite organization membership. The first owner is bootstrapped through a one-time Kubernetes secret or offline CLI, then the bootstrap path is disabled. Revocation invalidates active Deploy Admin sessions within five minutes.

7. Architecture and data

Browser→Private/Public Ingress→Deploy Admin API + UI→Postgres / Audit store
Deploy Admin API→Queue→Ephemeral K8s Job→infra-local or infra adapter→Target
  • API owns environments, policies, runs, stages, approvals, role bindings, allowlist entries, artifact references, and audit metadata.
  • Workers accept typed job specifications with a fixed command catalog; no arbitrary command endpoint exists.
  • Logs/artifacts use encrypted object storage and checksums; database stores indexed metadata and state transitions.
  • A lease with heartbeat protects each environment mutation. A lost worker marks the run reconciliation_required, never automatically failed-safe.
  • Outbound network policy allows only repository, registry, IdP, target cloud APIs, configured DNS/Cloudflare endpoints, and observability sinks.

Deployment run states

draft → queued → preflight → building → awaiting_approval → deploying → verifying → succeeded | failed | cancelled | reconciliation_required

8. YAML packaging contract

apiVersion: deploy.boxlite.dev/v1alpha1
kind: DeployAdmin
metadata:
  name: boxlite-deploy-admin
spec:
  image: ghcr.io/boxlite-ai/deploy-admin@sha256:<digest>
  ingress:
    mode: private
  auth:
    issuer: https://accounts.google.com
    clientIdSecretRef: deploy-admin-oidc
    requireAllowlist: true
  storage:
    databaseSecretRef: deploy-admin-postgres
  workers:
    serviceAccountName: deploy-admin-worker
    maxConcurrent: 3

The installation also defines separate API/worker service accounts, non-root security contexts, read-only root filesystems where possible, resource limits, PodDisruptionBudget, NetworkPolicy, encrypted persistent state, health probes, and migration/rollback instructions. Secrets use references; example YAML contains no values.

9. UX requirements

  • Overview answers: what is deploying, what failed, which environment is ready, and what needs approval.
  • The deployment wizard progressively discloses source, target, preflight, diff, and approval. Production impact is visually distinct.
  • Every mutating action shows target, source digest/SHA, expected effect, policy, and whether retry/rollback is safe.
  • Live logs are searchable, pausable, downloadable under permission, and preserve the stage boundary.
  • Keyboard navigation, visible focus, semantic labels, text plus color states, WCAG AA contrast, responsive layout, and UTC/local timestamps are required.

11. Success metrics and SLOs

  • ≥95% of routine staging releases initiated and completed without shell access within 60 days of rollout.
  • 100% of production releases have commit SHA, artifact digest, diff, approver, verification evidence, and audit correlation.
  • Median staging lead time ≤15 minutes excluding approval wait; deployment status freshness ≤10 seconds.
  • Unauthorized identities are denied before any environment metadata is returned.
  • Deploy Admin API target availability 99.9% monthly; recovery point objective 15 minutes and recovery time objective 60 minutes for management metadata. Target workloads remain independent of Deploy Admin availability.

12. Acceptance criteria

  1. Given a local worker and valid environment YAML, an Operator can run preflight, invoke the existing local adapter, follow logs, and see per-component verification.
  2. Given staging credentials, an Operator can select a commit, produce build evidence and SST diff, obtain required approval, deploy through the wrapper, and verify API/dashboard/Runner health.
  3. The same pinned Deploy Admin image installs on cloud K8s from declarative configuration and uses workload identity rather than embedded AWS keys.
  4. An authenticated Google user absent from the allowlist receives a generic denial and no environment data; adding/removing admission is audited.
  5. Production ingress can require VPN/private reachability in addition to Google SSO, allowlisting, RBAC, and two-person approval.
  6. Concurrent mutation of one environment is rejected; worker loss creates reconciliation-required state.
  7. Secrets are absent from browser payloads, logs, audit exports, YAML examples, and persisted command text.
  8. A plan containing protected Runner deletion/replacement cannot be approved through the standard release workflow.

14. Open decisions

  • Whether the local macOS worker remains the supported long-term local topology or L2 services are ported into Kubernetes.
  • Which CI artifacts can be promoted directly versus which SST assets must build inside the deploy Job.
  • Google Workspace domain restriction, session duration, and device posture requirements.
  • Audit retention and external SIEM destination.
  • Production approval quorum and emergency policy owners.
